前回の記事では、CLAUDE.md と SESSION.md の分離で autocompact 問題を解決する設計パターンを紹介した。

あれから20以上のプロジェクトをこの方式で実運用してきた。その過程で「こうすると劇的に良くなる」「これをやらないとハマる」というTipsがいくつも見つかった。

今回はそれを7つの「呪文」として紹介する。すべて実際の事故や気づきから生まれたものだ。

---

## 呪文1: 「〜を再開」で毎回新規セッションから始める

**長い会話を続けてはいけない。** 毎回新しいセッションを立ち上げて、「〇〇プロジェクトを再開」と一言言う。

```
ユーザー: 「アフターピルのプロジェクトに戻ります」
Claude: [CLAUDE.md → SESSION.md を読む → 現状把握 → 作業再開]
```

### なぜか

- autocompact のリスクがゼロになる（そもそもコンテキストが溜まらない）
- 新規セッションで SESSION.md を読み直すことで、**直前のセッションで更新した最新状態**から始まる
- 前のセッションで変な方向に行っていた場合のリセット効果もある

### 前提条件

**SESSION.md が常に最新であること。** これが成立していないと「再開」しても古い状態に戻ってしまう。前回の記事で紹介した自動更新プロトコル（タスク完了時・重要な判断時に SESSION.md を自動更新するルール）が必須。

### いつ切るか

- プロジェクトを切り替えるとき
- 同じプロジェクトでも、会話が長くなってきたら
- 作業の区切り（コミット後、大きな機能が完成した後）

## 呪文2: 「整合性、無矛盾性、効率性をチェック」

push 前にこの一言を言う。Claude がドキュメントとコードの齟齬を見つけて直してくれる。

```
ユーザー: 「整合性、無矛盾性、効率性をチェック。プッシュ。」
```

### 実際に見つかる問題

これは誇張ではなく、**ほぼ毎回何か見つかる**。実例:

| 見つかった問題 | 原因 |
|--------------|------|
| 「FAQ 4問」と書いてあるが実際は6問 | 2問追加した後、SESSION.md の古い記述を更新し忘れた |
| リポ公開状態が8件間違い | 作成時の思い込みで書いた（`gh repo view` で確認していなかった） |
| 見出し `## ユーザー preferences` が2回出現 | 編集でセクションを挿入した際に重複 |
| A→B→A の循環参照 | 「正本はあっち」を相互に書いた結果、正本がどこにもなかった |

### 「安全性」を足すタイミング

public リポを触っているときは「安全性」を追加する。

```
ユーザー: 「整合性、無矛盾性、効率性、安全性を深く検討してプッシュ」
```

Claude が個人情報・非公開リポ名・メールアドレス等が含まれていないかを `grep` でチェックしてくれる。

## 呪文3: 「深く検討して」

Claude はデフォルトで「さっさと答える」傾向がある。**「深く検討して」と明示的に言う**と、表面的な回答ではなく、トレードオフ分析・代替案の検討・エッジケースの考慮が出てくる。

### 実例: 機能の配置検討

あるサイトにFAQを追加する際の違い:

**「深く検討して」なし:**
```
Claude: FAQに追加しましょう。[即座にコード変更]
```

**「深く検討して」あり:**
```
Claude:
  配置候補A: FAQのみ — メリット/デメリット
  配置候補B: ヘッダー拡張 — メリット/デメリット
  配置候補C: フィルター周辺に注記 — メリット/デメリット
  配置候補D: 2列比較カード — メリット/デメリット
  配置候補E: FAQ + (?)アイコン — メリット/デメリット
  結論: Aが最適。理由は...
```

後者のほうが圧倒的に良い判断ができる。**正解が1つではない判断**（UI/UX、アーキテクチャ、フィルター設計など）で特に威力を発揮する。

## 呪文4: 決定事項のWHYを日付付きで記録

機能の追加・不採用を決めたら、**理由を含めて SESSION.md に記録する**。これをやらないと、次のセッション（あるいは数ヶ月後の自分）が同じ議論を繰り返す。

### 悪い例

```markdown
- 「現在営業中」フィルターは実装しない
```

### 良い例

```markdown
- 「現在営業中」フィルターは実装しない（2026-03-21）
  — 根拠が二次加工（パーサー推定、カバー率97.1%）で約290件が
  判定不能、パース不能薬局が最寄りの可能性があり代替なし、
  除外しなくても害がない（カード表示で十分）
```

### なぜ日付も入れるか

状況が変われば判断も変わる。日付がないと、永久に有効な不文律のようになってしまう。

### 実際に助かったケース

「駅名検索はいらないと前に決めなかったっけ？」と聞かれたとき、理由が書いてあれば即答できる。書いていなければ、同じ分析を最初からやり直しになる。

## 呪文5: 「競合と比べて何が負けているか」

新機能のアイデア出しに使える。Claude に競合サイトをフェッチさせて分析させる。

```
ユーザー: 「このサイトと比べて何が負けているか https://example.com」
```

### コツ: 「負けている」= 実装すべきではない

分析結果が出たら、**各項目について呪文3「深く検討して」を使う**。

実際の例では、競合サイトに3つの「負けている点」があったが、深く検討した結果:
- 1つは不要（既存機能で完全にカバー済み）
- 1つは不要（このサイトの責務ではない）
- 1つはFAQ追加のみで対応（大きな実装は不要）

「負けている」という直感を言語化し、本当に実装すべきかを呪文3で検証する。呪文3との合わせ技。

## 呪文6: フィードバックメモリでミスを構造的に防ぐ

Claude Code の memory 機能（`~/.claude/` 以下）に、**ミスの教訓を保存する**。同じミスをセッションをまたいで繰り返さなくなる。

### 書き方

```markdown
---
name: 既存リポの確認を怠らない
description: 新しいリポを作る前にリポ一覧を確認する
type: feedback
---

新しいリポの作成を提案する前に、リポ一覧を確認し、
既存リポで対応できないか検討すること。

**Why:** zenn-articles リポが既に存在するのに、
別のリポ内に articles/ を作ってしまった。

**How to apply:** 「新しく○○を作る」という発想が出たら、
まずリポ一覧を確認。
```

### 重要: 成功パターンも記録する

ミス（「やめろ」）だけでなく、**うまくいったアプローチも記録する**。修正ばかり記録すると、Claude が過度に慎重になって動きが悪くなる。

例: 「このリファクタでは1つのPRにまとめたのが正解だった」「この場面ではFAQ追加のみで十分だった」

## 呪文7: 正本を1つに決め、循環参照を潰す

情報の正本（single source of truth）を1つだけ決め、他は参照にする。

### 実際にやってしまった循環参照

```
CONVENTIONS.md: 「リポ一覧の正本は MEMORY.md」
MEMORY.md: 「リポ一覧は CONVENTIONS.md §9 が正本」
CONVENTIONS.md §9: 安全規則（リポ一覧じゃない）
```

**正本がどこにもない。** 結果、Zenn 記事用リポが既に存在するのに、別のリポ内に `articles/` ディレクトリを作ってしまった。

### 修正後

```
CONVENTIONS.md: 「リポ一覧の正本は MEMORY.md の
               『リポ一覧（正本）』セクション」
MEMORY.md → [実際のリポ一覧テーブルがここにある]
```

### コツ

- 参照先は**セクション名まで明記する**（「MEMORY.md」だけだと、どこにあるかわからない）
- 正本以外の場所で同じ情報を繰り返さない
- 参照先が本当に存在するか、呪文2の整合性チェックで自動的にカバーされる

## まとめ

| # | 呪文 | いつ使うか |
|---|------|-----------|
| 1 | 「〜を再開」 | セッション開始時（毎回） |
| 2 | 「整合性、無矛盾性、効率性をチェック」 | push 前（毎回） |
| 3 | 「深く検討して」 | 正解が1つでない判断 |
| 4 | WHYを日付付きで記録 | 機能の採用/不採用を決めたとき |
| 5 | 「何で負けてる？」 | 機能のアイデア出し |
| 6 | フィードバックメモリ | ミスが起きたとき / うまくいったとき |
| 7 | 正本を1つに | 情報の配置を設計するとき |

全部の根底にある思想は1つ: **Claude に「勝手にやらせる」のではなく、仕組みで品質を保証する**。

SESSION.md は自動更新の仕組み。整合性チェックは push 前の品質ゲート。フィードバックメモリは学習の仕組み。安全規則はガードレール。

Claude Code は強力だが、放っておくと雑な仕事をする。呪文（= 仕組み）で品質を構造的に担保する。そうすると、Claude Code は**信頼できるペアプロ相手**になる。
