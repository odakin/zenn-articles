## TL;DR

厚生労働省が公表する緊急避妊薬（アフターピル）の薬局一覧データ、約1万件。この営業時間フィールドが**完全な自由記述**で、一意なフォーマットが6,933種ある。これを構造化データにパースした話。最終カバー率97.1%。

**できたもの**: [緊急避妊薬 薬局検索](https://odakin.github.io/mhlw-ec-pharmacy-finder/)（GitHub Pages、無料、ソースも公開）

## 「まあ正規表現ちょっと書けばいけるやろ」

話はここから始まる。

厚労省の薬局データには `hours` という列がある。営業時間が入っている。こんな感じ。

```
月-金:9:00-18:00,土:9:00-13:00
```

よしよし。`,` で split して `:` で曜日と時間を分けて、`-` で range にすればいい。正規表現1本でいける。

**初版カバー率: 89.4%**

1割以上パースできなかった。なぜか？

## 地獄の門: 自由記述という名の無法地帯

このデータ、フォーマットの指定がない。各都道府県の各薬局が、好きなように入力している。同じ「月曜から金曜、9時から18時」を表すのに、こういうバリエーションが存在する:

```
月-金:9:00-18:00           ← きれいな形
月～金：９：００～１８：００  ← 全角まみれ
⽉-⾦:9:00-18:00           ← ？？？
月曜日～金曜日 9時～18時    ← 漢字時刻
月曜日~金曜日/9:00~18:00   ← スラッシュ区切り
(月火水木金)9:00-18:00     ← 括弧グルーピング
平日:9:00-18:00            ← 日本語略称
月から金は9時から18時       ← もはや散文
```

**全部同じ意味。**

これを「1つの正規形」に収束させるのが `normalizeHoursText()` の仕事。この関数は `.replace()` を **82回**呼ぶ。

## 第1の地獄: 見た目が同じなのに違う文字

```javascript
// 実データにある「月」の書き方
"月"  // U+6708 — 正しい
"⽉"  // U+2F47 — CJK互換文字
```

この2つ、見分けがつくだろうか。**つかない。** しかし正規表現 `[月火水木金土日]` は前者にしかマッチしない。

ある県のデータは全件が CJK互換文字 `⽉⽕⽔⽊⾦⼟⽇` で入力されていた。目視では正常に見えるのに、パーサーが全件スキップする。デバッグに半日かかった。

```javascript
// CJK互換文字を正規文字に変換
.replace(/⽉/g, "月").replace(/⽕/g, "火").replace(/⽔/g, "水")
.replace(/⽊/g, "木").replace(/⾦/g, "金").replace(/⼟/g, "土").replace(/⽇/g, "日")
```

## 第2の地獄: コロンが5種類ある

```javascript
"："  // U+FF1A 全角コロン（3,587件、全データの36%）
"∶"  // U+2236 RATIO
"︓"  // U+FE13 PRESENTATION FORM
"ː"  // U+02D0 MODIFIER LETTER TRIANGULAR COLON
":"  // U+003A 普通のコロン
```

全部コロンのつもりで入力されている。**全データの36%が「普通のコロン」以外を使っている。**

ダッシュに至っては**9種類**。

```javascript
"－" "‐" "−" "–" "—" "ー" "―" "ｰ" "-"
```

最後の `ー` は**カタカナの長音記号**。長音記号をハイフンの代わりに使う人がいる。

## 第3の地獄: 「金曜日曜」の恐怖

曜日を1文字に統一するため、`曜日` や `曜` を strip する処理がある。

```
月曜日-金曜日,日曜日
→ 月-金,日               ← 期待する出力
```

しかし `金曜日曜` という文字列（＝「金曜」と「日曜」がくっついたもの）がデータに存在する。

```
月曜-金曜日曜9:00-18:00
```

ナイーブに `曜日?` で strip すると:

```
月曜-金曜日曜        ← 入力
  ↓ 曜日? にマッチ
月 - 金              ← 「金曜日」がマッチして「日」まで食べた
```

**日曜が消えた。**

修正は先読みアサーション:

```javascript
.replace(/曜(?:日(?!曜))?/g, "")
```

`日` の後に `曜` が続く場合（= 次の「○曜」の先頭）は食べない。

しかしこれで strip した結果 `月-金日` になる。`金` と `日` がくっついてしまった。追加の修正:

```javascript
// range末尾と曜日がくっついたら「・」を挿入
// 「月-金日」→「月-金・日」
.replace(/([月火水木金土日])-([月火水木金土日])([月火水木金土日])/g, "$1-$2・$3")
```

1個のバグを直すたびに、新しいバグが生まれる。

## 第4の地獄: カンマの意味が3通りある

```
月-金:9:00-18:00,土:9:00-13:00    ← セグメント区切り
月-金:9:00-13:00,14:00-18:00      ← 午前/午後の2部制
月,水,金:9:00-18:00               ← 曜日の列挙
```

最初は `.split(",")` で分割していた。しかし2番目のパターンを split すると午前と午後が別のセグメントになり、午後に曜日がないのでパース失敗する。3番目を split すると `月` `水` `金:9:00-18:00` になり、`月` と `水` に時間がない。

**カンマの意味を文脈で判定しないといけない。**

```javascript
// ヒューリスティクス:
// - 「曜日+時間」→ 新しいセグメント
// - 「時間のみ」→ 直前のセグメントに追加（2部制）
// - 「曜日のみ」→ 次の要素とマージ（列挙）
```

## 第5の地獄: 曜日と時間の区切りが7種類

```
月-金:9:00-18:00     ← コロン（標準）
月-金 9:00-18:00     ← スペース
月-金9:00-18:00      ← 直接結合
(月火金)9:00-18:00   ← 括弧
月-金/9:00-18:00     ← スラッシュ
[月-金]9:00-18:00    ← 角括弧
月-金;9:00-18:00     ← セミコロン
```

セグメント間の区切りも同様に多彩: カンマ、スラッシュ、ピリオド、スペース、句点、直接結合。

## 第6の地獄: 時刻のタイプミス

```
9:00-8:00    ← 閉店が開店より前（おそらく18:00の誤り）
8:30-6:00    ← おそらく16:00か18:00
80:30-88:50  ← どうしてこうなった
```

タイプミスでもパースして表示すると、ユーザーが「8時閉店」を信じて行ってしまう。バリデーション（0:00〜29:59の範囲チェック）で弾き、生データ表示にフォールバックする。

ちなみに29:59まで許容するのは、日本の慣習で深夜営業を `25:00`（= 翌1:00）と表記するため。

## 最大の設計判断: 「間違いより欠落のほうがマシ」

ここまでの地獄を乗り越えて、カバー率が97%に達した。残り3%はどうするか？

答えは**「パースしない」**。生データをそのまま表示する。

なぜか。このサイトは**緊急避妊薬の薬局検索ツール**である。ユーザーは時間に追われている。「この薬局は土曜もやっている」と表示して、行ったらやっていなかったら、**取り返しがつかない**。

具体例:

```
月-金:9:00-18:00,土:???,日:9:00-14:00
```

土曜のデータがパース不能な場合、選択肢は2つ:

1. 土曜をスキップして月-金+日を表示 → ユーザーは「土曜は休み」と誤解するかもしれない
2. 全体をパース失敗にして生データ表示 → ユーザーが自分の目で判断できる

**2を選んだ。** 部分パースは返さない。全部パースできるか、全部失敗か。

同じ判断は「第N曜日」にも適用した:

```
第1・3土曜:9:00-12:00
```

これは「月の第1・第3土曜日だけ営業」の意味。しかしパーサーの出力は週次スケジュール（`{day: "土", open: "9:00", close: "12:00"}`）で、「第何週」を表現する型がない。

最初は「毎週土曜として表示」で実装した。しかし**それは嘘**だ。第2・4・5土曜に来局した人が無駄足になる。薬のアクセスに関わる情報で嘘をつくわけにはいかない。

修正後は第N曜日の情報を**丸ごとスキップ**する。土曜の情報が欠ける。しかし嘘よりはマシ。

## 祝日対応: もう1層の地獄

カバー率97.1%に到達した後、もう1つの問題に気づいた。

```
月-金:9:00-18:00,日祝休み
```

この薬局は祝日が休みだ。しかしパーサーは祝日を知らない。祝日の月曜に「営業中」と表示してしまう。

**祝日計算を丸ごと実装した。** 約60行、外部依存ゼロ。

- 固定祝日（元旦、建国記念日、天皇誕生日など）
- ハッピーマンデー（成人の日、海の日、敬老の日、スポーツの日）
- 春分・秋分（**天文公式で計算**、~2099年まで有効）
- 振替休日（日曜と重なったら翌月曜）
- 国民の休日（祝日に挟まれた平日）

```javascript
// 春分の日の天文公式
const vernal = Math.floor(
  20.8431 + 0.242194 * (year - 1980)
  - Math.floor((year - 1980) / 4)
);
```

祝日計算は純粋関数なのでテストしやすい。2026年の全18祝日（振替休日・国民の休日含む）が正確に計算できることを確認済み。

## カバー率の推移

```
89.4% → 95.7% → 96.6% → 96.3%(!) → 96.9% → 97.1%
```

96.6% → 96.3% で**下がっている**。これは「第N曜日を毎週として表示していた」嘘を修正して、スキップに変えたため。カバー率は下がったが、正確性は上がった。カバー率は品質の指標であって目標ではない。

## 残り2.9%はなぜ諦めたか

```
月～土：9:00~20：00、お客様感謝デーを除く日・祝：9:00~18:00
```

「お客様感謝デーを除く」。自然言語の条件。

```
月/水/金曜日第2/4火曜日第1/3/5土曜日9:00-18:00
```

曜日と序数と時刻が混然一体。

```
月	09:00	19:00
火	09:00	19:00
```

Excelからのコピペ。タブ区切り。

これらに個別対応するとコードが複雑化する。1パターン足すたびに既存の82個の `.replace()` との干渉をテストしないといけない。292件のために全9,951件のリグレッションリスクを負うのは割に合わない。

パース不能な薬局は生データをそのまま表示する。ユーザーは日本語が読めるので、問題ない。

## 教訓

1. **行政データの「自由記述フィールド」は地獄**。型を決めてくれ。`["月-金:9:00-18:00","土:9:00-13:00"]` みたいなJSONにしてくれ。頼む。

2. **正規化ルールには順序依存性がある**。1つ足すと別のが壊れる。82個を正しい順序で並べる作業は、もはやコンパイラの最適化パスに似ている。

3. **カバー率は品質の指標であって目標ではない**。96.3%に下がったコミットが一番大事な修正だった。

4. **健康に関わるデータでは「わからない」は「間違い」より100倍マシ**。パース不能なら生データを見せる。部分パースは嘘の温床。

5. **97%の壁を超えるコストは指数的に増える**。89→95は文字統一だけ。95→97はカンマの意味判定や先読みアサーション。97→99は自然言語処理の領域に片足を突っ込む。コストに見合わないと判断できることも技術力のうち。
